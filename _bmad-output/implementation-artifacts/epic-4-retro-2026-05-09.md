## What Went Well

### Learning Stayed Governed Instead of Decorative

The epic did not stop at generated summaries. Learning outputs became durable, inspectable artifacts and event payloads:

- Memory candidates are typed, confidence-scored, sensitivity-aware, and source-linked.
- Learning reviews contain structured facts, lessons, mistakes, missed assumptions, test gaps, memory candidate references, and skill signals.
- Retrospectives require retained evidence before producing review artifacts.
- Procedural learning outputs remain evidence-backed and candidate-first.

### Candidate-First Boundaries Held

The team repeatedly avoided silent promotion:

- Working memory stayed task/session scoped.
- Memory candidates require explicit review or policy-approved auto-save.
- Learning reviews can produce skill signals but not active skills.
- Procedural outputs are stored in learning-review artifacts, not skill registries.
- Epic 5 remains responsible for skill candidate aggregation, review UI, activation, promotion, and registry writes.

### Runtime Events Became a Strong Audit Spine

Epic 4 extended the event model without replacing it:

- `memory.candidate.created`
- `memory.entry.saved`
- `memory.candidate.reviewed`
- `learning.review.created`
- `memory.influence.recorded`
- `retrospective.review.created`

This preserved the architecture rule that CLI/TUI/RPC surfaces should render shared runtime truth rather than maintain separate learning state.

### Review Findings Converted Into Tests

Review cycles found real contract gaps and closed them with regressions:

- Story 4.1 tightened provider metadata sanitization, compacted resume command status, pending approval behavior, and working-memory refresh.
- Story 4.3 fixed safe reviewer metadata and action/status consistency.
- Story 4.5 fixed relevance gating, bounded retrieval, and sectioned learning-review previews.
- Story 4.6 fixed evidence-boundary gaps around terminal-event eligibility and storage revalidation.
- Story 4.7 tightened procedural influence references with `procout_` source IDs plus source session/task references.

### Epic 3 Lessons Were Applied

Epic 3 warned that memory and learning must be governed state, not hidden chat history. Epic 4 applied that lesson through provenance, source IDs, redaction, bounded previews, event history, and candidate-first lifecycle controls.

## Challenges and Growth Areas

### High-Blast-Radius Runtime Surfaces Remained Common

GitNexus repeatedly marked `AgentRuntime`, `validateRuntimeEvent`, `createTaskRequest`, `createFinalTaskSummary`, and memory/session readers as HIGH or CRITICAL impact. That was expected because learning crosses runtime, events, storage, context, and final summaries, but it forced disciplined additive changes and repeated full-suite validation.

### `detect_changes` Is Still Not Available

The GitNexus local CLI still lacks `detect_changes` / `detect-changes`. The team used `npx gitnexus analyze`, `npx gitnexus status`, symbol impact checks, scoped diffs, and validation as fallback evidence. This worked, but it remains a process gap.

### Lint Is Still Mostly Typecheck

`npm run lint` continues to delegate to TypeScript typechecking. This is useful, but it is not an independent static-quality or maintainability signal. The same risk appeared in prior retrospectives and remains unresolved.

### Learning Features Increase Context Pressure

Epic 4 added working memory, self-model state, memory candidates, durable memory, learning reviews, memory influence records, retrospectives, and procedural learning outputs. Context ordering, bounded previews, and secret filtering are now even more important because the system has more sources that can influence later tasks.

### Documentation Must Stay Implementation-First

The implementation artifacts are detailed and current. Public documentation still needs to stay concise and behavior-focused. The project should avoid putting roadmap or aspirational learning behavior in README-like docs before the behavior is implemented.

## Key Lessons

1. Learning output is only trustworthy when it preserves provenance and evidence references.
2. Candidate-first state is the safest bridge between runtime learning and future user-approved automation.
3. Memory reuse must be visible as `used`, `ignored`, or `contradicted`; hidden influence is unsafe.
4. Retrospectives need eligibility gates, not just a button.
5. Runtime events are product contracts, not incidental logs.
6. Review findings on learning/runtime boundaries should become regression tests.
7. Procedural memory should begin as skill-linked evidence, not active behavior.
8. Bounded previews and secret rejection must apply consistently across memory, events, summaries, artifacts, and final task output.

## Risks Entering Epic 5

### Skill Candidates Could Become Hidden Authority

Epic 5 must preserve the same candidate-first principle. A skill signal or skill candidate must not become an active skill unless the explicit promotion lifecycle approves it.

### Manual Skill Registry Needs Clear Trust Boundaries

Story 5.1 should define where manual skills live, what metadata is trusted, how invalid skill artifacts are handled, and how active registry entries differ from candidate outputs.

### Skill Invocation Can Affect Runtime Behavior

Story 5.2 changes execution behavior more directly than Epic 4 learning artifacts did. The runtime should record skill usage and keep current user instructions, repository policy, sandbox policy, and approval policy above skill guidance.

### Skill Usage and Influence Need Separate Semantics

Using a skill, being influenced by a skill signal, and promoting a skill candidate are different events. Epic 5 should avoid collapsing those concepts into one status field.

### Promotion Requires Safety Gates

Skill promotion must include evidence, trigger reason, risks, activation conditions, review state, and user approval. Promotion should use explicit state transitions and conflict checks.

## Significant Discovery Assessment

No Epic 4 discovery requires changing the Epic 5 roadmap.

Epic 5 story readiness should include these guardrails explicitly:

- Skill registry entries are manual/active artifacts; procedural learning outputs are candidate evidence.
- Skill invocation must be recorded separately from skill influence.

- Promotion requires explicit user approval and safe artifact validation.
- Runtime events should expose safe IDs/counts/summaries, not raw skill content or raw provider/tool output.
- Memory and skill influence must remain visible in task history and final summaries when materially relevant.

Epic update required: no.
