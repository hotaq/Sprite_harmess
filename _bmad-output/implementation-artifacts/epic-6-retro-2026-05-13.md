## What Went Well

### The TUI Stayed an Adapter

The biggest Epic 6 success is that TUI behavior stayed layered over shared runtime truth:

- `packages/tui` owns pure, deterministic view models and Ink rendering.
- `packages/cli` owns runtime/session/storage bridge behavior for live TUI usage.
- `packages/core` remains the owner of task lifecycle, final summaries, runtime events, approvals, policy, memory, and skill signals.
- React components do not read `.sprite` files or mutate runtime state directly.

This validates the Epic 5 retrospective guardrail: the UI can render and steer, but it must not become a second runtime.

### Review Findings Became Regression Tests

Epic 6 continued the review-before-done discipline:

- Story 6.1 review caught warning-classification drift in `createTuiRuntimeState()`.
- Story 6.5 review caught secret-like command headers and `/compact` visible-session fallback behavior.
- Story 6.6 review caught learning-review richness, final-summary event alignment, and CLI private-path redaction.

Each issue was patched with targeted regression coverage before the story moved to `done`.

### Prompt-First UX Survived Growing Feature Scope

The workbench gained message streams, approvals, slash commands, final summaries, and learning reviews without turning the UI into a noisy debug dashboard.

Important UX decisions held:

- Slash suggestions and input stay near the bottom prompt area.
- Runtime/context/details panels stay hidden unless requested.
- Outcome cards appear only when meaningful task terminal or learning events exist.
- Command and approval cards are bounded and redacted.

### Typed Runtime Events Became the Cross-Adapter Spine

The same runtime event model now supports:

- CLI output.
- Print/structured output.
- TUI activity stream.
- Live TUI controls.
- Final-summary cards.
- Learning-review cards.
- Future JSON-RPC event notifications.

This gives Epic 7 a stronger base than it would have had if Epic 6 had relied on assistant prose or UI-local state.

### RTK Command Discipline Is Now Explicit

Chinnaphat requested RTK-wrapped commands during review. That became the operating norm for validation and status checks:

- `rtk run npm test ...`
- `rtk run git diff --check && npm run lint && npm test`
- `rtk run npx gitnexus analyze ... && npx gitnexus status`

This reduces ambiguity in future story records and command evidence.

## Challenges and Growth Areas

### Redaction Is a Recurring Adapter Risk

Redaction issues appeared more than once:

- Story 6.5: unsupported slash command labels could expose secret-like values.
- Story 6.6: CLI slash result safe strings needed broader private-path redaction.
- TUI outcome panels needed both secret-like redaction and private absolute path redaction.

The lesson is that every adapter surface needs an explicit redaction fixture set. It is not enough to rely on one helper being used somewhere upstream.

### State Alignment Requires Extra Care

The TUI bridge has several sources of state: active runtime task, event history passed into live state construction, visible session IDs, command results, pending approvals, persisted learning artifacts, and prior TUI draft state.

Review found concrete alignment risks:

- `/compact` needed the visible TUI session when no explicit session ID was supplied.
- Final summary cards needed to use the same event list as the visible live state.
- Learning review cards needed artifact-backed details to satisfy the story, while renderer code stayed file-read free.

This pattern will matter even more in JSON-RPC, where multiple external clients may hold partial views.

### High-Blast-Radius Functions Needed Repeated Impact Checks

Functions such as `createTuiLiveWorkbenchState`, `TuiWorkbenchApp`, `handleLiveTuiInteraction`, `createCurrentLiveTuiState`, and outcome redaction helpers sit on broad adapter seams.

GitNexus impact analysis repeatedly flagged medium, high, or critical blast radius for seemingly small changes. The safe pattern was:

1. Run impact analysis before editing known symbols.
2. Announce high or critical risk.
3. Keep the patch narrow.
4. Add regression coverage in the affected adapter path.
5. Re-run targeted and full validation.

### `npm run lint` Still Mirrors Typecheck

The unresolved quality item from prior retrospectives remains: `npm run lint` is still a typecheck alias. Typechecking is valuable, but it is not an independent static-quality signal.

### GitNexus Detect-Changes Parity Still Needs Fallback Handling

AGENTS.md refers to `gitnexus_detect_changes()`, but the local CLI surface still exposes `analyze`, `status`, `query`, `context`, and `impact`, not a CLI `detect-changes` command. The working fallback remains:

- inspect `git diff` and staged scope,
- run targeted `gitnexus impact`,
- run forced `gitnexus analyze`,
- verify `gitnexus status`.

## Key Lessons

1. A TUI view model is allowed to summarize runtime truth; it must not create runtime truth.
2. Renderer components should be boring: render bounded safe state, do not read sessions, execute tools, or mutate memory/skills.
3. Adapter redaction must cover both secret-like values and private local paths.
4. Review findings are most valuable when converted into targeted tests immediately.
5. Prompt-first UX needs active protection as cards and panels accumulate.
6. Event-derived state is the strongest bridge from CLI/TUI to future RPC.
7. If a view uses both active task state and an event list, those inputs must be intentionally aligned.
8. Learning review display needs a safe bridge: rich enough to be useful, bounded enough to avoid leaking raw artifacts.

## Risks Entering Epic 7

### JSON-RPC Must Stay an Adapter

RPC must not create a second task loop, session store, approval engine, or event source. It should connect clients to `AgentRuntime` and shared storage/core contracts.

### Stdout Must Remain Protocol-Clean

Story 7.1 requires stdout for JSON-RPC responses/notifications and stderr for process diagnostics. Any accidental logs to stdout can corrupt the protocol stream.

### Protocol Shape Needs a Small Contract Before Implementation

Before implementing Story 7.1, define:

- JSON-RPC version envelope.
- Request/response/error shape.
- Notification naming pattern.
- Correlation/session/task metadata fields.
- How recoverable errors and next actions are represented.
- How startup/bootstrap state is exposed without secrets.

### Scope and Authorization Should Start Small but Be Explicit

Story 7.1 may only open the transport, but Epic 7 later depends on scoped permissions. The first transport shape should leave room for session/task scope checks rather than hard-coding global access assumptions.

### Do Not Import TUI for RPC Output

Epic 6 TUI read models are useful as design reference, but RPC should use core/runtime/storage contracts directly. RPC output should not depend on Ink, React, or `packages/tui`.

## Significant Discovery Assessment

No Epic 6 discovery requires changing the Epic 7 roadmap.

Epic 7 readiness should include these guardrails explicitly:

- JSON-RPC is an adapter over `AgentRuntime`, not a runtime replacement.
- stdout is reserved for protocol messages; diagnostics go to stderr.
- Event notifications are typed runtime events, not parsed assistant text.
- Final summaries and learning reviews should come from core/storage bridge contracts, not TUI renderer state.
- Approval responses must flow through shared runtime approval APIs.
- Redaction and bounded output rules must match CLI/TUI safety standards.
