## What Went Well

### Runtime-Owned Session Truth Held

Epic 3 successfully kept durable session truth below adapters. CLI commands render core/storage-owned data; they do not assemble session state or read compaction artifacts directly. This preserved the architecture rule that runtime, storage, and context assembly own durable task state.

### Append-Only History Became a Real Invariant

Story 3.1 established the key invariant: `events.ndjson` is canonical and append-only, while `state.json` is a recoverable snapshot. Later stories reused this model for inspection, resume, compaction, and compacted-context restoration instead of inventing separate state paths.

### Resume Stayed Conservative

Story 3.3 and Story 3.8 both reinforced the no-replay rule. Persisted validations, file edits, approvals, tools, provider calls, and compaction events are restored as evidence, not re-executed as side effects.

### Project Context Trust Boundary Was Clear

Story 3.4 treated `AGENTS.md`, `CLAUDE.md`, `SPRITE.md`, and `.cursorrules` as untrusted context. Repository guidance can inform the task, but it cannot override system/runtime policy.

### Compaction Moved from Artifact to Consumption

Stories 3.6, 3.7, and 3.8 formed a coherent compaction sequence:

1. Create structured compaction summaries.
2. Trigger manual compaction with a durable event.
3. Consume compacted context during resume with newer-event precedence.

This avoided the common failure mode where compaction creates an artifact that no runtime path actually uses.

### Heavy Reviews Improved Quality

Several stories benefited from adversarial review. The most important examples:

- Story 3.1 hardened persistence ordering, snapshot semantics, and storage append typing.
- Story 3.4 fixed bounded project-context reads and discriminated project-context record types.
- Story 3.6 added decision extraction, trusted active constraints, context-packet session validation, and read-consistency checks.
- Story 3.7 fixed duplicate artifact IDs and degraded snapshot-update warnings after durable compaction events.
- Story 3.8 fixed event-history precedence notes, bounded recent events, artifact session scope validation, and no-replay coverage.

## Challenges and Concerns

### `AgentRuntime` Remained a High-Blast-Radius Surface

GitNexus repeatedly flagged `AgentRuntime`, `createTaskRequest`, runtime event validation, and resume paths as HIGH or CRITICAL impact. This was expected because sessions and context continuity are cross-cutting, but it made narrow edits and full regression coverage mandatory.

### GitNexus `detect_changes` Was Still Unavailable

The local GitNexus CLI still lacks `detect_changes` / `detect-changes`. The team consistently recorded this limitation and used `gitnexus status`, re-indexing, symbol impact checks, scoped diffs, and full validation as fallback evidence.

### Review Findings Often Exposed Real Contract Gaps

Epic 3 review findings were not cosmetic. They often revealed contract ambiguity:

- Whether disk persistence should happen before in-memory publication.
- Whether `state.json` can be trusted over event history.
- Whether project context reads are truly bounded.
- Whether compaction summaries include the right continuity fields.
- Whether compacted context is bounded and precedence-aware.

The positive pattern is that the team converted these gaps into regression tests rather than merely updating prose.

### Static Quality Signal Is Still Thin

`npm run lint` still delegates to TypeScript typecheck. This remains useful as a baseline, but it is not an independent maintainability or style signal. The concern from prior retrospectives remains unresolved.

### Documentation Discipline Improved but Needs to Stay Active

README/progress cleanup improved during the epic. The risk is that future memory/learning work will again tempt the team to document aspirations instead of implemented behavior. Public docs should remain concise and current; detailed progress belongs in implementation artifacts.

## Key Lessons

1. Durable state is safety-sensitive state.
2. Append-only event history should remain the source of truth; snapshots are recovery aids.
3. Resume must restore evidence, not replay authority.
4. Context sources need explicit trust levels, ordering, and redaction behavior.
5. Compaction is only useful when restore paths consume it with bounded recent evidence.
6. Newer state and events must supersede stale compacted summaries, and that precedence must be visible.
7. Review findings on runtime contracts should usually become regression tests.
8. GitNexus impact checks are valuable even when they only confirm that edits are risky.

## Risks Entering Epic 4

### Memory Can Become Unsafe or Noisy

Epic 4 will turn session evidence into memory candidates and possibly durable memory entries. Without provenance, confidence, sensitivity filtering, and review gates, memory can become both unsafe and misleading.

### Learning Must Not Become Hidden Authority

Prior memory or lessons may influence future tasks, but the influence must be visible and auditable. Memory cannot silently override current user instructions, runtime policy, or repository evidence.

### Self-Model Must Not Overclaim

Story 4.1 needs the runtime self-model to identify actual tools, provider state, sandbox state, context state, memory state, and loaded skills. It must not claim unavailable capabilities.

### Context Budget Pressure Will Increase

Epic 3 added session and compacted context. Epic 4 adds working memory, durable memory, lessons, and skill signals. Context source ordering and bounded previews become even more important.

### Review Workflow Becomes Product Behavior

Memory candidate review/edit/reject is not internal housekeeping. It is product functionality and must have clear states, user authority, audit events, and redaction.

## Significant Discovery Assessment

No discovery from Epic 3 requires changing the Epic 4 plan.

The plan remains sound, but Epic 4 should make several guardrails explicit before implementation:

- Memory candidates are candidate-first; durable promotion requires policy and review controls.
- Memory entries require provenance, confidence, type, timestamp, source task/session, and sensitivity state.
- Raw logs, raw tool output, secrets, credentials, private keys, tokens, and large code chunks are excluded by default.
- Prior memory influence must be recorded as `used`, `ignored`, or `contradicted` when materially relevant.
- Self-model context must be factual and bounded.

Epic update required: no.

Epic 4 story readiness update required: yes, each story should include these guardrails explicitly.
