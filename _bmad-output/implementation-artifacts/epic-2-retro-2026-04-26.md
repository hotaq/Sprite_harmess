## What Went Well

### Safety-First Capability Sequence

The story order worked well. Each story added one safety layer before the next capability depended on it:

1. Inspect files safely.
2. Track file activity.
3. Apply targeted patches.
4. Classify risk.
5. Execute commands through a sandbox.
6. Require approval for risky actions.
7. Run configured validation.
8. Record recovery from failures and denials.
9. Prevent unsafe memory persistence.

That progression avoided jumping directly from file access to shell execution without policy, approval, validation, and recovery controls.

### Runtime-Owned Truth Stayed Intact

The team kept core task truth in `AgentRuntime` and canonical runtime events rather than moving state into adapters or README examples. This was visible across tool lifecycle events, file activity, policy decisions, approvals, validation events, recovery events, and memory safety events.

### Metadata-Only Audit Discipline Improved

Epic 2 repeatedly strengthened the rule that audit events should describe what happened without storing raw repository content, command output, patch bodies, or secret-looking values. Review findings in Stories 2.2, 2.4, 2.8, and 2.9 turned this from a principle into tested behavior.

### Review Findings Became Regression Tests

The team did not just mark review findings as resolved. Several findings became explicit tests:

- Filesystem race and inaccessible path handling.
- Multibyte output truncation.
- Secret-looking file activity summaries.
- Malformed file edit metadata.
- Configured validation command risk classification.
- Sandboxed timeout handling.
- Approval action/type constraints.
- Pending approval gates.
- Recovery linkage validation.
- Raw output alias rejection.
- Memory candidate safety evaluation.

### Documentation Was Cleaned Up After the Epic

README was kept focused on setup and current behavior, while detailed progress moved into `progress.md`. That helps future users onboard without reading the full internal story history.

## Challenges and Concerns

### Quality Remained the Main Risk

The Epic 1 retrospective identified quality as the main risk entering Epic 2. That was accurate. Epic 2 repeatedly touched shared runtime and event contracts, and several stories needed review-driven fixes before being marked done.

This is not a blame issue. The system was moving into safety-sensitive behavior, so hidden edge cases were expected. The useful lesson is that safety work needs review, targeted tests, and GitNexus blast-radius checks before claiming completion.

### Cross-Cutting Runtime Changes Had High Blast Radius

GitNexus impact analysis flagged critical or high risk around `AgentRuntime`, runtime event validation, final summaries, and safety/memory integration. Those warnings were appropriate because small contract changes can affect many execution flows.

Future work should keep high-blast-radius edits narrow, additive when possible, and backed by targeted plus full validation.

### Approval Editing Had a Contract Nuance

Story 2.6 surfaced an important distinction: shared approval response data can describe a modified request, but runtime file-edit execution actually needs the executable modified tool call shape. The team resolved this by making `modifiedToolCall` the official runtime/shared contract for modified `apply_patch` approvals.

That distinction should carry into UI, RPC, and TUI approval flows later.

### `npm run lint` Is Still Not a Separate Signal

`npm run lint` remains a `typecheck` alias. This keeps the baseline simple, but it still does not provide independent static style or maintainability checks. The concern from Epic 1 remains open.

### Documentation Needed a Post-Epic Cleanup Pass

README accumulated progress-level detail while Epic 2 was moving quickly. The later split into README plus `progress.md` fixed the immediate issue, but future stories should keep public docs concise during implementation, not only after completion.

## Key Lessons

1. Runtime-owned state and events must remain the source of truth for safety-sensitive behavior.
2. Audit payloads should be metadata-only by default; raw content needs a strong reason and a safe storage boundary.
3. Approval edit contracts must preserve executable runtime shape, not only user-facing request descriptions.
4. GitNexus blast-radius checks are useful early warning signals, especially for `AgentRuntime` and event schema edits.
5. Deny, timeout, malformed input, and recovery paths deserve first-class tests, not only success-path coverage.
6. README should document current user behavior, while story-level progress belongs in implementation artifacts or `progress.md`.

## Risks Entering Epic 3

### Durable State Raises the Safety Bar

Epic 2 safety decisions were mostly runtime-local. Epic 3 introduces durable session artifacts such as `events.ndjson` and `state.json`. Anything accidentally persisted can survive process restarts, so redaction and metadata-only discipline become even more important.

### Event Log and State Snapshot Must Not Drift

Story 3.1 says `events.ndjson` is append-only and `state.json` is recoverable but not the only source of truth. The implementation must keep that invariant clear so resume behavior does not silently trust stale snapshots over newer events.

### Session State Can Re-Expose Sensitive Context

Story 3.2 will display session ID, cwd, task goal, plan, recent events, files touched, commands run, pending approvals, last error, and next step. Every displayed field should be considered potentially sensitive until safety filters prove otherwise.

### Project Context Is Untrusted

Story 3.4 loads `SPRITE.md`, `AGENTS.md`, `CLAUDE.md`, or `.cursorrules`. That should build on Epic 2's policy precedence lesson: repository-provided instructions are useful context, not higher-priority authority.
