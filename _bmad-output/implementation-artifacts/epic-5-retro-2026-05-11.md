## What Went Well

### Candidate-First Skill Evolution Held

The epic preserved the core safety model from Epic 4: learning can propose evidence-backed artifacts, but it cannot silently modify future behavior.

- Skill signals remain `signal_only`.
- Skill candidates are stored outside active skill registry roots.
- Candidate generation writes reviewable artifacts, not active skills.
- Promotion writes a manual project skill only after explicit confirmation.
- Rejected and draft candidates remain inspectable without becoming task guidance.

### Runtime Events Became a Stronger Product Contract

Epic 5 added skill lifecycle events without making adapters own runtime truth:

- `skill.invoked`
- `skill.invocation.failed`
- `skill.usage.recorded`
- `skill.signal.recorded`
- `skill.candidate.created`
- `skill.candidate.skipped`
- `skill.candidate.reviewed`

These events now give CLI, future TUI, future RPC, audit history, learning review, final summaries, and tests one shared source of truth.

### Review Findings Became Regression Tests

The strongest pattern across the epic was review-driven hardening:

- Story 5.1 review found metadata and path-redaction gaps in skill listing.
- Story 5.2 review found unsafe-manifest/content policy-blocking gaps in manual invocation.
- Story 5.3 review hardened skill usage text bounds and raw path rejection.
- Story 5.4 heavy review caught fail-open raw signal normalization and signal-shape drift.
- Story 5.6 heavy review caught symlink promotion, lifecycle invariants, CLI edit reachability, and promoted skill evidence gaps.
- Story 5.7 review caught candidate influence bypasses, leak checks, resume/future-task coverage gaps, source-qualified candidate references, embedded candidate IDs, and promoted manual skill suggestion false positives.

The team repeatedly converted these findings into deterministic tests instead of relying on reviewer memory.

### Prior Research Improved the Design

Hermes Agent and related research were useful as design pressure, not copy targets.

- Hermes inspired skill lifecycle and curation thinking.
- The implementation deliberately avoided active skill writes from raw signals.
- Story 5.5 used a candidate-only pipeline rather than immediate skill generation.
- Story 5.6 made promotion an explicit review state transition.

### GitNexus and Impact Discipline Reduced Runtime Risk

Skill features touched high-blast-radius surfaces such as `AgentRuntime`, `validateRuntimeEvent`, skill registry APIs, final summaries, runtime loop tests, and CLI smoke tests. GitNexus impact checks, stale-index repair, and full validation were consistently used before commits.

## Challenges and Growth Areas

### Runtime and Event Surfaces Remain High Blast Radius

Skill lifecycle work crosses registry scanning, task context assembly, event validation, learning reviews, session persistence, final summaries, CLI output, and candidate storage. Even narrow behavior changes can affect many flows.

The safe pattern is now clear:

1. Name the exact functions/contracts before editing.
2. Run GitNexus impact or document the fallback.
3. Add a red regression.
4. Keep the implementation additive or tightly scoped.
5. Run targeted tests, then full validation.

### Candidate Safety Needed More Negative Coverage Than Expected

The hardest risks were not happy-path promotion. They were negative paths:

- candidate IDs masquerading as skill references,
- source-qualified candidate references,
- stored candidate names appearing in ignored usage suggestions,
- raw candidate body or routing hints leaking through failure payloads,
- resume/future-task paths reintroducing candidate guidance,
- promoted candidate identity being confused with promoted manual skill identity.

Story 5.7 showed that separation is a system property, not a single registry check.

### `npm run lint` Still Mirrors Typecheck

`npm run lint` still delegates to TypeScript typechecking. Typecheck is valuable, but it is not an independent maintainability or style signal. This remains an unresolved quality-improvement item from the Epic 4 retrospective.

### GitNexus `detect_changes` Parity Is Still Missing

The local GitNexus CLI supports `analyze`, `status`, `query`, `context`, and `impact`, but not the `detect_changes` command referenced by AGENTS.md. The team used forced analyze/status plus diffs and validation as fallback evidence.

### Public Documentation Must Stay Behavior-First

The team cleaned progress detail out of README earlier. Epic 5 added real user-facing skill behavior, so public docs can now describe implemented commands and lifecycle boundaries. They should still avoid aspirational claims about autonomous skill evolution.

## Key Lessons

1. A skill signal is evidence, not authority.
2. A skill candidate is a review artifact, not an active skill.
3. Promotion must be explicit, audited, and separated from candidate creation.
4. Skill usage, skill influence, skill signals, and skill promotion need distinct event semantics.
5. Safety reviews are strongest when every finding becomes a focused regression.
6. Negative-path tests matter more than happy-path tests for self-evolving behavior.
7. Runtime events are now the bridge to TUI and RPC; they must stay safe, typed, and adapter-neutral.
8. Bounded review metadata is useful; raw candidate content, paths, secrets, routing rules, diffs, and patches must stay out of public outputs.

## Risks Entering Epic 6

### TUI Must Stay an Adapter

Epic 6 must not create a second task loop. The TUI should render runtime events and send typed user intents back to `AgentRuntime`.

### Runtime Event Coverage Must Be Mapped Before Rendering

Story 6.1 and Story 6.2 should map which existing events and state APIs power each visible panel before introducing UI state reducers.

### Approval and Steering Must Not Bypass Runtime Policy

Story 6.3 must route approval responses, cancellation, and steering through runtime APIs. The TUI must not execute commands or apply edits directly.

### Secret Redaction Is Now Broader

The TUI will display memory, skills, model/provider state, validation, tool output, and final summaries. It must preserve the same redaction rules already enforced in CLI/runtime outputs.

### TUI Tests Should Exercise Core Without Launching a Full UI

Architecture says core runtime logic must remain testable without launching the TUI. Epic 6 should keep adapter-local rendering tests separate from runtime contract tests.
